# Converse-Framework feedback (Test-run-plan build)

- The WebSocket recipe was useful as a starting point, but a production-ish app still has to duplicate transport/session glue to support settings, status, and runtime provider rebuilds. A small reusable `WebSocketSession` helper with hooks for app-level messages would make framework trials faster and reduce copy/paste drift.
- Runtime provider changes currently require app code to rebuild a bundle and recreate the collector because the collector owns the VAD provider and the pipeline owns the provider bundle. A documented public reload/update pattern would make settings UIs safer to implement.
- `PocketTTSProvider` exposes `set_quantize`, but voice changes require rebuilding the provider. A first-class `set_voice`/`configure` method returning status would fit the advertised voice-selection capability.
- Browser clients need to hand-roll PCM capture, resampling, frame slicing, and playback. The existing JS audio player is helpful, but a matching browser microphone/frame sender module would make web speech-to-speech demos much easier to stand up.
- `FasterWhisperASRProvider.transcribe_audio` emitted the queued progress event and then hit an empty-message `AssertionError` because `_transcribe_blocking` assumed `_model` was already loaded. Providers that lazy-load should consistently call their ensure/load path inside first-use execution methods.
- `SpeechPipeline.handle_audio_turn` serializes `str(exc)` for `asr.error`; empty-message exceptions become unhelpful UI events like `asr.error`. Including the exception class or a fallback message would make integration debugging much faster.
- On Windows, the `nvidia-cublas-cu12` wheel installed `cublas64_12.dll` under `site-packages\nvidia\cublas\bin`, but CTranslate2 did not find it unless the app/launcher added NVIDIA wheel `bin` folders to the DLL search path. Framework docs or a small helper for CUDA DLL discovery would make faster-whisper CUDA setup less brittle for app authors.
- The app had to hard-code Pocket TTS voice names. A provider-level `list_voices()` or structured `ProviderStatus.voices` field would let settings UIs stay accurate as `pocket-tts` adds, removes, or renames voices.
- `ProviderBundle.check_statuses()` is convenient, but provider status checks have uneven side effects: some are cheap import/HTTP probes, while others may load heavy models. Splitting this into `probe_status()` vs. `load_status()` semantics would make startup diagnostics more predictable.
- The framework emits useful progress events, but provider setup failures still arrive as generic stage events plus later errors. A standard `provider.error` / `provider.loaded` event shape across VAD, ASR, LLM, and TTS would make UIs easier to build and debug.
- The browser WebSocket path needs HTTPS or a tunnel for mobile microphone access. A framework demo launcher or documented Cloudflare/ngrok/local-network recipe would help users test real mobile-first speech flows without each app reinventing that setup.
